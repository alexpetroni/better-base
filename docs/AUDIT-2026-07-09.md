# Adversarial evaluation — better-base (2026-07-09)

Six parallel specialist reviews (security, resilience/ops, architecture, data model,
simplification, frontend) reading the real code under `apps/web/`. Findings are grouped
by root cause; severity is the reviewer's, adjusted where multiple reviews converged.
Every item cites `file:line`. Nothing here was auto-fixed.

Overall: the codebase is genuinely well-built — DI services, Result-typed errors, thin
routes, integer-cents money, additive migrations, signature-verified webhooks, an
idempotent email sender, server-only secret boundaries, real progressive enhancement,
zero `any`. The issues below are the gap between "works in a demo" and "survives
concurrency, failure, and two years of edits."

---

## Theme A — Non-atomic read-modify-write (systemic; 3 sites, one root cause)

Every counter does `read state → compute in JS → write absolute value`, which is not
atomic. It both **over-admits** under concurrency and (worse) **loses increments** so a
racing client bypasses the cap.

- **Login rate limiter** — `auth/rate-limit.ts:36-43` + `admin/login/+page.server.ts:30-46`.
  N concurrent login POSTs all read the same count → ~`5 × concurrency` password guesses
  per window. (security H1)
- **Chat rate limiter** — `chat/service.ts:102-110` + `chat/rate-limit.ts:57-64`. Same
  flaw; concurrent burst exceeds 20/hr. (resilience #5)
- Both are also **fixed-window**, allowing the 5-at-end + 5-at-start boundary burst.

**Fix once:** a single shared atomic upsert — `INSERT … ON CONFLICT DO UPDATE SET
count = table.count + 1` with the window reset in SQL, enforced via `RETURNING`. This
also resolves the duplication finding (two 65-line copies of the same fixed-window logic,
simplification #4).

## Theme B — No transactions, anywhere

`grep '.transaction(' lib/` returns nothing.

- **Order creation is the dangerous one** — `shop/webhook.ts:83-142`. Order row,
  order_items, stock decrement, and email are 4+ independent awaits. If items/stock throw
  after the order commits (transient DB error, deadlock, kill), the order has claimed
  `stripeSessionId`, so Stripe's redelivery hits `onConflictDoNothing` → returns
  "duplicate" → does nothing. Net: **customer charged, order has zero line items, stock
  never decremented, no email, permanently unrecoverable.** (resilience #1, HIGH)

**Fix:** wrap order + items + stock in one `db.transaction()`; send the (idempotent)
email after commit.

## Theme C — No timeouts / no pool limits (silent hangs, not graceful failure)

- **DB pool unbounded** — `db/client.ts:9` `new pg.Pool({connectionString})` with no
  `max`, `connectionTimeoutMillis`, `idleTimeoutMillis`, or `statement_timeout`. pg
  defaults to 10 conns and waits forever; a few hung requests starve everything including
  `/api/health`. (resilience #2, HIGH)
- **No timeout on any outbound call** — Resend fetch (`email/resend.ts:14`), Anthropic
  (`chat/anthropic-provider.ts:14,18`), Stripe (`shop/stripe-gateway.ts:10`). A hung
  socket pins the request and its DB connection. In the webhook the email is awaited
  inline (`webhook.ts:129`), so a Resend hang pins the whole webhook. (resilience #3, HIGH)
- **Chat SSE never aborts on client disconnect** — `api/chat/+server.ts:63-79` has no
  `cancel()`; the Anthropic stream runs to completion (billed tokens) after the tab
  closes, then writes to a dead request. (resilience #4, HIGH)

**Fix:** set pool `max`/timeouts + server `statement_timeout`; add `AbortSignal.timeout`
to Resend and `timeout`/`maxRetries` to the Stripe/Anthropic clients; thread an
`AbortController` into the SSE `cancel()`.

## Theme D — The cross-site content bundle duplicates schema knowledge and is already lossy

This is the **highest-value single fix**: the export/import CLI is the entire point of
the two-DB design, and it silently drops columns.

- `content/bundle.ts:25-83` hand-redeclares `ArticleContent`/`QuizContent`/
  `ProductContent`/`MediaDescriptor` as parallel copies of the Drizzle row types, with
  hand-written field-by-field mappers in `export.ts`/`import.ts`. Four places must change
  in lockstep with **zero compiler enforcement**. (architecture #2 HIGH, data-model HIGH-1)
- **Already broken:** `media.blurhash` exists (`media/schema.ts:26`) but is absent from
  the descriptor and mapper — every imported image loses its blur-up placeholder today.
- Add any column later (e.g. `products.sku`) → export omits it, import ignores it, content
  silently diverges between sites, discovered in production.

**Fix:** derive the `*Content` types from `typeof table.$inferSelect` via `Pick`/`Omit`
so a schema change fails to compile until the bundle is updated; add a round-trip parity
test per type; add `blurhash` now.

Related import gaps: an item whose pillar slugs don't exist in the target is created with
**zero tags → invisible** in listings, with only a stderr warning and a zero exit code
(data-model MED-2); re-import never refreshes changed bytes for a reused key and orphans
unreferenced media (data-model MED-3).

## Theme E — Missing indexes on foreign keys (Postgres does not auto-index them)

Real code paths do seq scans; each user/media/product delete is O(table).

- `quizzes.pillar_id` (`quiz/schema.ts:18`) — joined on the public quiz path. (data HIGH-2)
- `quiz_results.subscriber_id`, `orders.email`, `orders.stripe_payment_intent` — all hit
  by GDPR erase and the refund webhook. (data MED-1)
- `order_items.product_id`, `articles.cover_media_id`, `products.cover_media_id`,
  `articles.created_by`, `quizzes.created_by`, `media.created_by` — FKs with
  `ON DELETE set null` and no index.

**Fix:** one additive migration adding a covering index per FK column. No redundant
indexes were found — the gap is only missing ones.

## Theme F — Public endpoints that trigger emails are unthrottled (abuse / quota burn)

- Newsletter signup (`newsletter/+page.server.ts:11-39`) and quiz-result email
  (`quiz/[slug]/rezultat/[resultId]/+page.server.ts:22-37`) accept an attacker-supplied
  email and send with no per-IP limit and no CAPTCHA. Idempotency only suppresses repeats
  to the *same* address, so distinct victims each get a fresh send → mailbomb
  amplification, burned Resend reputation, unbounded `subscribers` rows. (security H2, HIGH)

**Fix:** per-IP + global rate limit (reuse the Theme-A shared limiter) + CAPTCHA/PoW.

## Theme G — Module-boundary rule is unenforced theater

- `eslint.config.js:40-48` restricts only the `$lib/modules/*/*` **alias**; modules import
  each other by **relative path**, which the rule never matches. Routes/components are
  clean, but the modules themselves are a mesh: `quiz/funnel.ts` → crm/email internals,
  `gdpr/erase.ts` reaches into 5 modules, `content/*` into 4, `email/templates.ts` →
  `shop/money.ts`, etc. A dev told "edit freely behind the barrel" will break three
  modules by renaming an internal. (architecture #1, CRITICAL-to-maintainability)

**Fix:** either enforce the relative-parent import ban (`eslint-plugin-boundaries` or a
`no-restricted-imports` pattern on `../*/`), or drop the pretense and document that
modules share schema/services. Today it's neither. Cross-module *schema* imports (FK
relations) are unavoidable in one shared DB; the avoidable ones are the service/logic
imports the barrels already expose.

## Other notable findings

### Security (beyond A/B/F above)
- **M1** SVG uploads served verbatim — stored-XSS surface if opened cross-origin
  (`media/validation.ts:12`). Drop svg or set `IMGPROXY_SANITIZE_SVG=true` + attachment
  disposition.
- **M2** Rate-limit keys on `getClientAddress()` with no `ADDRESS_HEADER`/`XFF_DEPTH`
  configured — behind a proxy all IPs collapse to one bucket, and a naive later fix makes
  XFF spoofable. Document the proxy hop depth.
- **H3 (conditional)** Committed default imgproxy key/salt (`docker-compose.yml:48-49`,
  `.env.example:47-48`) with `:-<default>` fallbacks — if any deploy inherits them,
  signing is forgeable. Drop the fallbacks so a missing secret fails fast; placeholders
  only in `.env.example`.
- **L5** `BETTER_AUTH_SECRET` doubles as the HMAC secret for consent + chat tokens — one
  leak compromises all three. Add a dedicated `TOKEN_SECRET`.
- **L1–L7** body-read-before-size-check, tokens in logged URL paths on 500, media confirm
  not bound to the presign ticket, no persistent Stripe event-id ledger, no imgproxy
  source-size caps. (see per-item fixes in the security review)

**Verified strong:** no auth catch-all (limiter can't be sidestepped), admin guard is
truly server-side for all `/admin/*` incl. actions and endpoints, role not client-settable,
signup disabled, Stripe prices always server-looked-up (no client tampering), chat session
ownership via `timingSafeEqual` HMAC, no `sql.unsafe`/injection anywhere, no secret logged.

### Resilience (beyond B/C above)
- **#6** `chat_rate_limits` grows unbounded — prune deletes only sessions
  (`chat/service.ts:156-166`). Delete expired `ip:` rows in the same cron.
- **#7** Overselling the last unit under concurrent checkout — stock checked at checkout,
  decremented only post-payment with no hold (`checkout.ts:108`, `webhook.ts:124`).
  Reserve at checkout or detect+refund in the webhook txn, or document as accepted.
- **#8** Quiz submission not idempotent — double-submit creates duplicate result rows
  (`quiz/[slug]/submit/+server.ts:26`).
- **#9** `/api/health` throws 500 (not 503) when env is unset (`health/+server.ts:9`).
- **#10** Fail-fast incomplete — only chat validates env at boot; `DATABASE_URL`,
  `BETTER_AUTH_SECRET`, `S3_*`, and `RESEND_API_KEY` (when `EMAIL_DRYRUN=false`) resolve
  lazily → app boots healthy then 500s on first use. Validate all required env at startup.

**Verified strong:** email idempotency is race-safe (claim-by-insert + guarded error
retry), order idempotency handles concurrent Stripe delivery, migrations additive-only,
health checks bounded by `Promise.race`, structured error logging with per-error id.

### Data model (beyond D/E above)
- **HIGH-3** Media referenced from `products.gallery` (jsonb) and inline `media:` markdown
  refs has **no referential integrity** — deleting media dangles them silently. Add a
  pre-delete scan (`gallery @> [id]`, markdown `LIKE`), or a real `product_media` join
  table for galleries.
- **LOW-2** Email uniqueness is case-sensitive `text UNIQUE`; dedup works only because the
  service lowercases. Any bypassing writer breaks it. Use `citext` or `LOWER(email)` unique
  index.
- **LOW-1** Quiz uses a single nullable `pillar_id` while articles/products use join tables
  — asymmetric; the bundle carries `pillars[]` but quiz import uses only `[0]`.
- **LOW-4** `updated_at` is app-managed with no trigger — a forgetful update path leaves it
  stale, and import "changed?" detection leans on it.

**Verified strong:** integer cents everywhere, all `timestamptz`, join tables cascade
correctly, `media_kind_shape` CHECK constraint, idempotency uniques correct.

### Simplification (highest code savings first)
- Triplicated `ensureUniqueSlug`/`slugTaken` (blog/shop/quiz service) → one generic. ~45 lines.
- Duplicated pillar validate+replace + `pillarSlugsFor` (blog vs shop) → shared
  `setPillars`/`pillarSlugsFor`. ~35 lines.
- Route form-action boilerplate — identical `create` action 4×, `failOf` mapper 4×,
  `sitePillars` map 4×, `String(form.get())` ~40×, list-filter idiom 3× → small helpers +
  a `createEntityAction` factory. **Largest aggregate saving.**
- Fixed-window rate-limit logic duplicated (resolves with Theme A).
- Per-module `Result` envelope copy-pasted 7× → one generic `Result<T,E>`; keep per-module
  `E` unions.
- `slug.ts`/`money.ts` trapped inside blog/shop but imported by pages/quiz/email → promote
  to shared `$lib/util`.
- Date formatting inline 9× → one `formatDate` helper (also fixes frontend #13 TZ bug).
- Dead code: `isPurchasable` (0 callers), `media/server.ts:16-23` raw-primitive
  re-exports, `createImgUrl`, several unused const re-exports; the public unguarded
  `/dev/form` demo page.
- Hidden shared state: `registerMediaReferenceCheck` global array populated by import
  side-effects (`media/service.ts:34`) — order-dependent boot; a tree-shake silently
  disables media-delete protection. Pass checks in via `MediaDeps` instead.

### Frontend (a11y/SEO/perf)
- Chat streaming has no `aria-live` region — invisible to screen readers
  (`ChatPanel.svelte:82-100`).
- Home page ships only `<title>` — no description/canonical/OG (`(public)/+page.svelte:8`).
- Chat force-scrolls to bottom on every token, hijacking a reader scrolled up
  (`ChatPanel.svelte:19-22`).
- Quiz result pages (PII scores) are indexable — no `noindex`
  (`.../rezultat/[resultId]`).
- `srcset` is DPR-only, so the `sizes` prop is dead and retina over-fetches ~2× on the
  product cover (`imgproxy.ts:68-70`). Emit width descriptors.
- Sitemap omits products and CMS pages (`sitemap.xml/+server.ts:15`).
- No `hreflang` alternates despite ro/en (`+layout.svelte:32-39` uses a display:none hack).
- Chat mid-stream error orphans a half-written reply with no retry.
- Cookie banner overlaps the chat widget (both fixed bottom `z-50`).
- Date formatting SSR/client TZ mismatch (hydration) — pin `timeZone`.
- Public forms/admin login lack pending/disabled state → double-submit.
- One hardcoded RO `aria-label` bypasses i18n (`(public)/+layout.svelte:57`).

**Verified strong:** clean Svelte 5 runes throughout, genuine progressive enhancement,
`<Img>` alt contract enforced + server-only signing, JSON-LD escaped, consent checkboxes
default-unticked, system-font stack (no FOIT), ro/en message parity 275/275.

## Suggested priority order

**P0 — correctness/safety under real traffic**
1. Wrap order creation in a transaction (Theme B).
2. Make rate limiters atomic + add throttling to public email endpoints (Themes A, F).
3. DB pool limits + outbound timeouts + SSE abort (Theme C).
4. Fix the content bundle to derive from schema + add `blurhash` + parity test (Theme D).

**P1 — resilience/data integrity**
5. FK indexes migration (Theme E).
6. Fail-fast env validation + health 503 (resilience #9/#10).
7. Media reference integrity on delete; imgproxy key fallbacks removed; dedicated
   `TOKEN_SECRET`; drop/​sanitize SVG.
8. Import "all pillars skipped" → hard failure; quiz idempotency; unbounded rate-limit
   table pruning.

**P2 — maintainability/UX**
9. Enforce or drop the module-boundary rule (Theme G).
10. Extract the duplication (slug, pillars, Result, form-action factory, date, rate-limit);
    promote shared utils; delete dead code.
11. Frontend a11y/SEO/perf batch (aria-live, home SEO, result noindex, width srcset,
    sitemap, hreflang, chat scroll/error UX, double-submit guards).
